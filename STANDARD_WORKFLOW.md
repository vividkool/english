# 共通開発ガイドライン (Common Development Guidelines)

このドキュメントは、複数のプロジェクトで共通して適用される、AIと共創するための標準的な開発フローとセキュリティルールを定義します。

## 1. セキュリティと機密情報管理 (Secrets Management)

- **環境変数の徹底**: APIキー、シークレット、認証情報は絶対にソースコードに直接記述せず、`.env` ファイル（ローカル）または GitHub Secrets（CI/CD）を利用します。
- **`.env.example` の提供**: 各リポジトリには必ず環境変数のテンプレートファイルを用意し、セットアップを容易にします。
- **履歴の保護**: 万が一機密情報がプッシュされた場合は、履歴の初期化（パージ）を迅速に行います。

## 2. 開発サイクル: GitHub Issue 起点の自動実装

タスクの管理と実装はすべて GitHub Issue を起点とします。

### STEP 1: タスクの投稿 (ユーザー)
- タイトル: `YYYY-MM-DD` または 簡潔な機能名
- 内容: 実装・修正したい内容を詳細に記載。
- この Issue が作成・更新されると、GitHub Actions が自動的に起動します。

### STEP 2: AIによる自動実装と検証 (Actions)
- AIスクリプトが Issue 内容を解釈し、ソースコードと `todo.md` を自動更新します。
- 変更内容は `develop` ブランチへ自動的にコミット・プッシュされます。

### STEP 3: プレビューとフィードバック
- `develop` ブランチへのプッシュにより、自動的にプレビュー環境（Firebase Hosting 等の `develop` チャネル）へデプロイされます。
- ユーザーはプレビューを確認し、さらなる修正があれば Issue に追記します。

## 3. ブランチ戦略とリリースフロー

- **`develop` ブランチ**: 
  - AIによる実装、プレビューデプロイが行われる動的なブランチです。
- **`main` ブランチ**: 
  - カスタムドメインが紐付く、安定した本番環境です。
  - `develop` での動作確認後、**手動**でプルリクエストを作成しマージします。
- **本番デプロイ**: 
  - 事故を防止するため、本番環境への最終デプロイは原則として**ローカル環境から手動**で行います。

## 4. ドキュメント運用
- **`todo.md`**: 現在進行中のタスクと直近の予定を管理。
- **`changed.md`**: 完了したタスクの履歴を移動。`todo.md` の軽量化を図ります。
