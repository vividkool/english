# 標準開発ワークフロー (Standard Development Workflow)

## 1. セキュリティ第一の機密情報管理 (Secrets Management)

機密情報の漏洩を防ぐため、以下のルールを厳守します。

- **`.env` のプッシュ禁止**: 
  - 絶対に `.env` ファイルを同期（プッシュ）しないでください。
  - プロジェクトに必要な環境変数は `.env.example` にテンプレートとして記述します。
- **ハードコードの禁止**: 
  - APIキーやシークレットは、コード内に直接記述せず必ず環境変数を経由させます。
- **Service Account キーの扱い**:
  - `service_account.json` 等のファイルは原則リポジトリに含めず、`.gitignore` に追加されていることを常に確認してください。
- **履歴漏洩時の対応**:
  - 万が一キーを履歴に含めてプッシュしてしまった場合は、即座にキーを失効（Rotate）させ、リポジトリの履歴を消去（リセット）してください。

## 2. 開発サイクル: GitHub Issue を起点とした自動実装

日中と夜間のライフサイクルを分けた「AI共創フロー」を採用します。

### STEP 1: タスクの投稿 (ユーザー)
ユーザーがスマホ等から GitHub Issue を作成します。
- タイトル: `YYYY-MM-DD` 形式の日付
- 内容: 実装したい機能、修正したいバグの具体的な説明。

### STEP 2: AIによる自動実装 (AI)
Issueをトリガーに AI が起動し、以下の処理を自動で行います。
- `develop` ブランチでのコード変更。
- `todo.md` の更新。
- 定期的な `todo.md` から `changed.md` への完了タスク移行（読み込み省力化のため）。

### STEP 3: プレビューとフィードバック
- プレビュー環境への自動デプロイ後、Issue にコメントが届きます。
- ユーザーはプレビューを確認し、必要に応じて Issue や PR で指示を追加します。

## 3. ファイル運用のルール

- **情報の集約**:
  - 現在のタスクは `todo.md` に集約。
  - 完了したタスクの履歴は `changed.md` に移動し、`todo.md` の肥大化を防ぐ。
- **低負荷・高パフォーマンス**:
  - UIスレッドをブロックする重い処理は避ける。
  - 秘密鍵の検証などはバックエンド/ランタイム環境でセキュアに実行する。
- **最新状態の維持**:
  - AIが作業を開始する前に必ず `git pull` を行い、ローカルとリモートを同期する。
  - セッション終了時には必ず `git push` を行う。
